agent-manager · State Detection & Presentation derived from lib/state.sh, lib/hooks/state-hook.sh, lib/status-bar, cmd/am-browse, internal/sessions/

State of every am session is resolved on-demand by a single canonical resolver (_state_resolve). The resolver merges four signals: tmux pane process tree, agent-fired hook files, ANSI-stripped pane content, and Claude conversation JSONL. Two consumers render the result: the bash status-bar that paints tmux tabs, and the Go am-browse TUI that lists sessions for the picker.

Signal sources

Hook files (push)
lib/hooks/state-hook.sh

Claude Code / Codex fire lifecycle hooks. Script maps event → am state and writes one of waiting_input | waiting_permission | waiting_custom | running to $AM_STATE_DIR/<session>.

Also persists Claude conversation UUID to a per-session sidecar $AM_STATE_DIR/<session>.sid so the resolver can target the right JSONL without racing on shared sessions.json.

tmux pane process tree
ps + tmux display-message / list-panes

Top pane PID + descendants drive the shell-vs-agent distinction and the search for a live claude child (used to resolve session UUID when sidecar misses).

Pane content (pull)
tmux capture-pane → ANSI-strip → 40 lines

Regex pass detects permission prompts (Claude & Codex), plan-approval / /ask, Codex Working (Xs) spinner, Claude (running) / ❯ prompt.

Claude JSONL (pull, Claude only)
~/.claude/projects/<encoded>/<sid>.jsonl

Resolver tails the file, finds last assistant/user/queue-operation entry, inspects stop_reason. Stale-gate 30s. Conversation chosen via sidecar → pane --session-id argv → lsof; only falls back to newest-mtime when no signal resolves.

Presenters

tmux tab strip (bulk, hot path)
lib/status-bar

One tmux list-panes + one ps -eo + one jq build the bulk fixtures SESSION_TOP_PID / PROC_COMM / PROC_CHILDREN + REG_*. For each session calls _state_resolve by nameref — no per-session forks.

Writes three options per session in one batched tmux set-option ; ... chain so every session has its own pre-rendered "current" view (instant on switch):

  • @am_status_left — clickable strip (status-left)
  • @am_sidebar — compact strip (pane-border-format)
  • @am_attention — counter of non-current waits (status-right)

Refresh trigger: 5s tmux status-interval tick, plus refresh-client -S issued by every hook fire.

Session picker (Go TUI)
cmd/am-browse/main.go

Bubbletea program. Reads sessions via internal/sessions.LoadEntries → ListTmuxSessions, ReapOrphans, RefreshTitles, ReadRegistry. Sorts by activity desc, renders list + preview pane (preview-cmd = lib/preview).

Stdout protocol: session name → attach, __NEW__, __RESTORE__, or empty → cancel. Falls back to lib/fzf.sh when binary not built.

Browser itself does not compute am-state; live state lives in the tmux tab strip painted by status-bar. The browser shows the registry task field, refreshed by RefreshTitles.

List helper
cmd/am-list-internal

Same LoadEntries path; prints session|display lines for legacy fzf reload and am list [--json].

Session resolved in order: $AM_SESSION_NAME → tmux #S of $TMUX_PANE → registry cwd match. Race guard skips running writes when current state file is waiting_* unless event is UserPromptSubmit (only user action transitions waiting → running). Hook also writes .sid sidecar, invalidates .list_cache and .title_scan_last, and issues tmux refresh-client -S.

Cache & coordination files

$AM_STATE_DIR/<session>
One-line state file written by state-hook.sh. Read by _state_hook_read. Removed by ReapOrphans.
$AM_STATE_DIR/<session>.sid
Claude conversation UUID sidecar. Per-session writer avoids the shared-sessions.json read-modify-write race when concurrent hooks fire.
$AM_DIR/sessions.json
Registry. Holds directory / branch / agent_type / task per session. Read by both bash and Go (ReadRegistry); atomic temp-rename writes.
$AM_DIR/.gc_last
60s throttle shared by bash registry_gc and Go ReapOrphans. Whichever runs first stamps the marker.
$AM_DIR/.title_scan_last
60s throttle shared by bash auto_title_scan and Go RefreshTitles. Hooks invalidate it on prompt boundaries so the next status-bar tick refreshes within ~5s.
$AM_DIR/.list_cache
Cached am list output. Hooks delete it so the next fzf reload picks up the new state.
$AM_DIR/.state-debug.log
Opt-in trace: AM_STATE_DEBUG=1 writes one line per resolution showing which step won (shell|hook|pane|jsonl|fallback|classify_exit).
$AM_DIR/.hook-debug.log
Opt-in trace: AM_HOOK_DEBUG=1 writes one line when a hook fires but exits without writing state (registry miss, no session resolvable, cwd inaccessible).

Invariants worth remembering

  • Single resolver. Both agent_get_state (per-session) and status-bar (bulk tick) call _state_resolve. Bulk fixtures are passed by bash 4.3+ nameref; without them the resolver builds size-1 fixtures inline.
  • Hook is primary; pane is corrective. Stage 2 short-circuits on waiting_*, but stage 3 still runs for running so a permission prompt painted during a slow tool call is caught.
  • Per-session sidecar. Claude conversation UUID lives in <session>.sid rather than the shared registry — eliminates the read-modify-write race on sessions.json when concurrent hooks fire.
  • Status-bar pre-renders one strip per perspective. Every session's @am_status_left already has itself marked current, so the tmux switch reads a precomputed option — no async #() wait on hot path.
  • Throttles are file-marker based and cross-language. .gc_last / .title_scan_last coordinate bash and Go without IPC.
  • Browser presents tasks, not state. The Go TUI shows registry data; live am-state appears only via the tmux tab strip painted by status-bar.
